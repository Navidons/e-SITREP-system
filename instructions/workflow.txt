### 1. Executive Summary

The **SITREP Automation System** is a secure, scalable, and professional web-based enterprise platform designed to fully automate the creation, submission, review, approval, and dissemination of **Daily and Weekly Situation Reports** from over 60 border posts in Uganda.

The system eliminates manual compilation errors, reduces turnaround time from hours to minutes, ensures data integrity, and provides real-time visibility for decision-makers while maintaining strict compliance with government data classification (RESTRICTED).

**Key Outcomes:**
- Accurate, consistent, and timely SITREPs
- Strong auditability and accountability
- Role-based access control across Stations, HQ, and Admin levels
- Seamless export of station-level, consolidated daily, and weekly reports
- Foundation for future integration with PISCES, MIDAS, and other NCIC systems

---

### 2. Business Requirements & Current Process

**Current Manual Process:**
- Each border post submits a daily report (e.g., Busia format) by 1400HRS.
- HQ manually compiles all reports into one consolidated tabular Daily SITREP.
- Weekly reports are manually aggregated in Excel (stations × days).

**Automated Requirements:**
- Station officers (Inputters) enter data for their station only.
- Multi-level review workflow (Reviewer → Verifier → Authoriser).
- One-click generation of:
  - Individual Station Daily Report
  - Consolidated National Daily SITREP (exact current format)
  - Weekly Excel Summary + Totals
- Full audit trail and role-based permissions.

### 3. System Architecture (High-Level)

**Architecture Style:** Microservices-ready, API-first, Layered Architecture

**Core Layers:**
- **Presentation Layer**: Responsive Web Portal (PWA)
- **Application Layer**: Business Logic & Workflow Engine
- **Data Layer**: PostgreSQL with Row Level Security
- **Integration Layer**: Future APIs for PISCES/MIDAS
- **Reporting Layer**: PDF + Excel generation engine
- **Security Layer**: Authentication, Authorization, Audit

**Deployment Model:** Containerized (Docker), On-premise first with hybrid cloud backup.

### 4. Database Design (PostgreSQL)

#### Core Tables

```sql
-- Master Data
CREATE TABLE border_stations (
    id SERIAL PRIMARY KEY,
    code VARCHAR(20) UNIQUE,
    name VARCHAR(100) UNIQUE NOT NULL,
    cluster VARCHAR(50),
    type VARCHAR(30),
    active BOOLEAN DEFAULT true
);

-- Users & RBAC
CREATE TABLE users (...);           -- See RBAC section
CREATE TABLE roles (...);
CREATE TABLE user_roles (...);
CREATE TABLE permissions (...);
CREATE TABLE role_permissions (...);

-- Daily Reports
CREATE TABLE station_daily_reports (
    id SERIAL PRIMARY KEY,
    station_id INTEGER REFERENCES border_stations,
    report_date DATE NOT NULL,
    submitted_by INTEGER REFERENCES users,
    submission_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status VARCHAR(20) DEFAULT 'draft', -- draft, submitted, reviewed, verified, approved, rejected
    staff_on_duty INTEGER,
    medical_screening TEXT,
    general_remarks TEXT,
    urgent_matters TEXT,
    UNIQUE(station_id, report_date)
);

-- Movements (Arrivals & Departures)
CREATE TABLE movements (
    id SERIAL PRIMARY KEY,
    report_id INTEGER REFERENCES station_daily_reports,
    movement_type VARCHAR(10) CHECK (movement_type IN ('arrival', 'departure')),
    nationality_code VARCHAR(10),
    male INTEGER DEFAULT 0,
    female INTEGER DEFAULT 0,
    total INTEGER GENERATED ALWAYS AS (male + female) STORED
);

-- Special Categories & Incidents
CREATE TABLE special_categories (...);
CREATE TABLE incidents (...);

-- Audit Logs
CREATE TABLE audit_logs (...);
```

**Key Features:**
- Materialized views for fast aggregation
- Row Level Security (RLS) policies
- Triggers for automatic audit logging

### 5. User Roles & RBAC (Enterprise Standard)

| Role                    | Level               | Key Permissions |
|-------------------------|---------------------|-----------------|
| **ADMIN**               | System              | User/Role management, full exports, system config |
| **HQ_AUTHORISER**       | Headquarters        | Final approval, generate & disseminate consolidated SITREP |
| **HQ_VERIFIER**         | Headquarters        | Data verification |
| **HQ_REVIEWER**         | Headquarters        | Review & comment |
| **CLUSTER_SUPERVISOR**  | Cluster             | Review stations under cluster |
| **STATION_INPUTTER**    | Border Post         | Data entry & own station report export only |

**Implementation:**
- Fine-grained permissions stored in database
- JWT-based authentication with role claims
- PostgreSQL Row Level Security (RLS) for data isolation
- Admin portal for role assignment

### 7. Business Workflow

1. **Data Entry** → Station Inputter creates/submits report
2. **Review** → HQ Reviewer checks and comments
3. **Verification** → HQ Verifier validates numbers
4. **Approval** → HQ Authoriser approves
5. **Generation** → System creates official Consolidated SITREP
6. **Dissemination** → PDF ready for distribution
7. **Weekly Roll-up** → Automatic aggregation every Sunday/Monday

All transitions are logged in audit trail.
